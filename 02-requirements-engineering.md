# Requirements Engineering Fundamentals

Requirements engineering is the process of discovering, documenting, analysing, validating, and managing the needs and constraints of a software system.

## Functional Requirements

Functional requirements describe what the system must do.

Examples include:

- The user shall be able to create an account.
- The system shall calculate the total price of an order.
- The administrator shall be able to remove users.
- The system shall send a confirmation email after registration.

## Non-Functional Requirements

Non-functional requirements describe the quality attributes and constraints of the system.

Common categories include:

- Performance
- Security
- Reliability
- Usability
- Availability
- Compatibility
- Maintainability
- Scalability

Example:

The system shall display search results within two seconds.

## User Requirements

User requirements describe system services and constraints in language understandable to customers and end users.

They are usually written at a high level and avoid unnecessary technical details.

## System Requirements

System requirements provide a detailed and technical description of system behaviour, interfaces, constraints, and functionality.

They are mainly used by developers, testers, architects, and other technical stakeholders.

## Requirements Elicitation

Requirements elicitation is the process of collecting information from stakeholders.

Common elicitation techniques include:

- Interviews
- Questionnaires
- Observation
- Workshops
- Brainstorming
- Document analysis
- Prototyping
- Focus groups

## Requirements Analysis

Requirements analysis evaluates collected requirements to identify:

- Conflicts
- Missing information
- Ambiguity
- Duplication
- Technical feasibility
- Business priority
- Dependencies

## Requirements Specification

Requirements specification documents the agreed requirements.

A Software Requirements Specification, or SRS, commonly contains:

- Project purpose
- Scope
- Functional requirements
- Non-functional requirements
- External interface requirements
- Constraints
- Assumptions
- Acceptance criteria

## Requirements Validation

Requirements validation checks whether the documented requirements correctly represent stakeholder needs.

Validation techniques include:

- Reviews
- Inspections
- Prototypes
- Test-case creation
- Stakeholder meetings
- Consistency checks

A good requirement should be:

- Correct
- Complete
- Clear
- Consistent
- Feasible
- Testable
- Traceable
- Prioritised

## Requirements Traceability

Requirements traceability connects each requirement to its source, design components, implementation, and test cases.

A requirements traceability matrix can help determine whether every requirement has been implemented and tested.

## Requirements Management

Requirements management controls changes to requirements throughout the software development lifecycle.

It includes:

- Version control
- Change requests
- Impact analysis
- Prioritisation
- Approval
- Traceability
- Status tracking
