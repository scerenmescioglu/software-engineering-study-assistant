# Software Project Management Fundamentals

Software project management is the process of planning, organising, monitoring, and controlling software-development work.

Its purpose is to deliver a suitable software product within agreed constraints.

## Project Constraints

Software projects are commonly affected by:

- Scope
- Time
- Cost
- Quality
- Resources
- Risk

A change in one constraint may affect the others.

For example, increasing project scope may require more time, money, or staff.

## Project Scope

Project scope defines what the project will and will not include.

A clear scope helps prevent uncontrolled expansion of requirements.

## Scope Creep

Scope creep is the uncontrolled addition of features or requirements after a project has started.

It can cause:

- Delays
- Increased cost
- Reduced quality
- Team overload
- Missed objectives

Scope changes should be evaluated and approved through a controlled process.

## Stakeholders

A stakeholder is a person, group, or organisation affected by or able to influence a project.

Examples include:

- Customers
- End users
- Developers
- Testers
- Project managers
- Sponsors
- Government or regulatory bodies
- Support teams

## Project Planning

Project planning may include:

- Defining objectives
- Identifying deliverables
- Estimating effort
- Estimating cost
- Creating a schedule
- Assigning responsibilities
- Identifying risks
- Defining quality expectations
- Selecting tools and technologies

## Deliverable

A deliverable is a measurable project output.

Examples include:

- Requirements specification
- Design document
- Prototype
- Source code
- Test report
- User manual
- Deployed application

## Milestone

A milestone is an important point or event in the project schedule.

Examples include:

- Requirements approved
- Design completed
- Prototype demonstrated
- Testing completed
- Product released

A milestone normally has no duration of its own.

## Work Breakdown Structure

A Work Breakdown Structure, or WBS, divides the project into smaller and more manageable pieces.

A WBS helps teams:

- Understand the required work
- Estimate effort
- Assign responsibilities
- Track progress
- Identify missing tasks

## Task Dependency

A task dependency describes how one task relies on another.

Common dependency types include:

- Finish-to-start
- Start-to-start
- Finish-to-finish
- Start-to-finish

Finish-to-start is the most common dependency.

It means one task must finish before another task can begin.

## Gantt Chart

A Gantt chart displays project activities across a timeline.

It may show:

- Task names
- Start dates
- End dates
- Durations
- Dependencies
- Milestones
- Progress

## Critical Path

The critical path is the longest sequence of dependent activities that determines the minimum project duration.

A delay in a critical-path activity can delay the entire project.

Critical-path tasks normally have little or no scheduling flexibility.

## Estimation

Software estimation predicts the effort, time, cost, and resources needed for a project.

Estimation methods include:

- Expert judgement
- Analogy
- Planning poker
- Function points
- Use-case points
- Story points
- Algorithmic estimation models

Estimates should be reviewed when requirements or assumptions change.

## Story Points

Story points are a relative measure used by Agile teams to estimate work.

They may consider:

- Complexity
- Effort
- Uncertainty
- Risk

Story points are not equivalent to hours.

## Velocity

Velocity is the amount of work completed by an Agile team during an iteration.

It is commonly measured using completed story points.

Velocity should mainly be used for planning by the same team rather than comparing different teams.

## Risk

A project risk is an uncertain event that may affect project objectives.

A risk has:

- Probability
- Impact

Typical software-project risks include:

- Changing requirements
- Staff turnover
- New or unfamiliar technology
- Security vulnerabilities
- Unrealistic deadlines
- Poor communication
- Insufficient testing
- Dependency failure
- Budget reduction

## Risk Management

Risk management includes:

1. Risk identification
2. Risk analysis
3. Risk prioritisation
4. Response planning
5. Monitoring and control

## Risk Responses

Common risk responses include:

- Avoid: remove the cause of the risk
- Mitigate: reduce probability or impact
- Transfer: move responsibility to another party
- Accept: acknowledge the risk and prepare if necessary

## Risk Register

A risk register records information such as:

- Risk description
- Cause
- Probability
- Impact
- Priority
- Owner
- Planned response
- Current status

## Project Roles

### Project Manager

The project manager may:

- Plan the project
- Coordinate activities
- Manage resources
- Communicate with stakeholders
- Monitor progress
- Manage risks
- Resolve issues
- Control scope and schedule

### Business Analyst

The business analyst helps identify, analyse, and document stakeholder requirements.

### Software Developer

The developer designs, implements, reviews, and maintains software.

### Software Tester

The tester evaluates software quality and identifies defects.

### Product Owner

The Product Owner represents product needs and prioritises the product backlog in Scrum.

### Scrum Master

The Scrum Master supports the Scrum process and helps remove obstacles.

## Communication Management

Good communication helps ensure that stakeholders understand:

- Project status
- Current risks
- Decisions
- Responsibilities
- Changes
- Deadlines
- Problems requiring attention

Communication methods may include:

- Meetings
- Reports
- Dashboards
- Email
- Documentation
- Issue trackers
- Team messaging tools

## Change Management

A change request should be evaluated before implementation.

Impact analysis may consider:

- Cost
- Schedule
- Architecture
- Security
- Testing
- Resources
- Existing requirements
- User documentation

Approved changes should be documented and traceable.

## Quality Management

Software quality management may include:

- Defining standards
- Conducting reviews
- Testing
- Auditing processes
- Tracking defects
- Measuring quality
- Performing continuous improvement

## Configuration Management

Configuration management controls project items such as:

- Source code
- Requirements
- Design documents
- Test cases
- Build files
- Releases

Version control is an important configuration-management tool.

## Project Monitoring

A project manager may monitor:

- Completed work
- Remaining work
- Schedule variance
- Cost variance
- Defect counts
- Risks
- Team capacity
- Change requests
- Milestone status

## Project Closure

Project closure may include:

- Confirming deliverable acceptance
- Completing documentation
- Releasing resources
- Archiving project files
- Recording lessons learned
- Closing contracts
- Preparing a final report

## Lessons Learned

Lessons learned record what worked well, what caused difficulties, and what should be improved in future projects.

They support organisational learning and better planning.
